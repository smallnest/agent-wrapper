Design Decisions

Decision Moved RunInput from root package to types

Ambiguity: RunInput was in the root package alongside the Agent interface, but Agent.Run() takes RunInput as a parameter. The claude/ package needs to call Run() but also needs to be registered in the root Registry.

Choice: Moved RunInput to types package alongside Session, Message, and Event.

Rationale: The root package's Registry needs to import provider packages (e.g., claude/) for registration. But claude/agent.go must reference RunInput in its Run() signature. If RunInput stays in root, importing root from claude/ creates a cycle. Moving it to types breaks the cycle naturally — types has no import of root or provider packages.

Decision Separate RegisterIn() function instead of auto-registration

Ambiguity: The spec implied providers are available via Registry.Get("claude-code") out of the box. But importing claude/ from root to register causes an import cycle.

Choice: claude/register.go exports RegisterIn(r *Registry) that callers invoke explicitly. The registry ships with stubs that return "not yet implemented".

Rationale: This avoids the import cycle and follows Go's convention of explicit registration (similar to database/sql driver registration). Callers opt in: claude.RegisterIn(registry). The stub preserves the discoverability benefit (List() shows all providers).

Decision Binary auto-detect searches PATH first, then home directories

Ambiguity: The acceptance criteria say "优先 PATH，其次 ~/.local/bin，其次 ~/.npm-global/bin" but didn't specify whether BinaryPath overrides auto-detect.

Choice: If BinaryPath is set, return it directly (no auto-detect). Otherwise search PATH → ~/.local/bin → ~/.npm-global/bin. Result is cached per agent instance via sync.Once.

Rationale: Explicit BinaryPath should be honored without any filesystem checks. Auto-detect is expensive (multiple LookPath calls) and the result won't change during the agent's lifetime.

Deviations

Deviation Stdin is closed immediately after sending both requests

Spec said: JSON-RPC requests sent via stdin, implying stdin stays open during the session.

Implemented: Stdin is closed right after writing initialize + run requests. The subprocess reads stdin until EOF, then continues writing to stdout.

Why: Claude Code's agent mode is a single-shot protocol: send inputs, receive streaming events. Keeping stdin open would require the goroutine to hold a pipe indefinitely. Closing stdin lets the subprocess know no more input is coming and avoids resource leaks.

Tradeoffs

Tradeoff Shell script mocks vs. Go-based mock process

Alternatives: (A) Shell scripts in temp dirs that echo JSON-RPC lines (current), (B) Go test helper that builds and runs a tiny binary, (C) Interface-based mock with fake stdin/stdout.

Pros/Cons: (A) is simplest, portable, no build step. (B) is more realistic but slower (compilation). (C) skips the subprocess layer entirely and wouldn't test the real JSON-RPC scanning pipeline.

Winner: (A) — the tests verify the full pipeline (subprocess → scanner → parseNotification → Event) with minimal setup cost.

Open Questions

Question Is the stdin close-after-send pattern correct for Claude Code's real CLI?

The implementation closes stdin immediately after sending both JSON-RPC requests. This assumes Claude Code doesn't need further stdin input during the run. If the real CLI expects to read additional requests (e.g., for multi-turn within a single process), this will cause early termination.

Verify: Test with the real claude agent binary to confirm the single-shot stdin pattern works.

Question Should parseNotification handle unknown notification methods?

Currently, unknown notification methods are silently skipped (returns false). If Claude Code adds new notification types in future versions, they'll be dropped without any logging or error.

Verify: Decide whether to log unknown methods or forward them as generic events for extensibility.
