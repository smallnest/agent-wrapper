Implementation Notes

Issue #2 — [#4] Process Manager — 子进程 + Scanner — 2026-06-02

Design Decisions

Decision Constructor function instead of Start() method

Ambiguity: The issue spec mentioned Start() as a method on agentProcess, but didn't specify the API surface precisely.

Choice: Used a package-level StartProcess(ctx, ProcessConfig) constructor function returning *AgentProcess.

Rationale: Constructor functions are idiomatic Go (e.g., os.Open, net.Dial). A separate Start() method would require a two-phase init (zero-value struct then start), which invites misuse.

Decision Shared Frame.Data and Frame.Raw for JSONRPC frames

Ambiguity: The spec defines Data []byte and Raw []byte as separate fields, but for line-delimited JSON they contain identical content.

Choice: Frame{Data: b, Raw: b} — both fields point to the same allocation.

Rationale: Halves per-frame allocations. The fields serve different semantic purposes (Data = parsed payload, Raw = debugging), but for JSONRPC frames the bytes are identical. SSE frames legitimately differ (Raw includes the data: prefix reconstruction).

Decision ProtocolError omits Provider field from scanner

Ambiguity: types.ProtocolError has a Provider field, but the scanner doesn't know which provider it's scanning for.

Choice: Scanner produces ProtocolError with empty Provider. The agent implementations (claude/codex/pi/opencode) that own the provider identity will wrap or annotate the error.

Rationale: Separation of concerns — the process/scanner layer is protocol-agnostic; provider identity belongs to the agent layer. Avoids threading provider through the scanner API.

Deviations

Deviation Context cancellation uses exec.Command + goroutine, not exec.CommandContext

Spec said: Context-based graceful shutdown: ctx cancel → SIGTERM → wait 5s → SIGKILL.

Implemented: Used exec.Command (not CommandContext) with a dedicated context-monitor goroutine that triggers the SIGTERM→SIGKILL sequence.

Why: exec.CommandContext sends SIGKILL immediately on cancel, bypassing the graceful SIGTERM step. By managing signals ourselves, we enforce the spec's shutdown sequence. A sync.Once ensures idempotency between Close() calls and context cancellation.

Deviation SSE Raw field is reconstructed, not original bytes

Spec said: Raw []byte = "raw line text (for debugging)".

Implemented: SSE Raw is reconstructed as "data: " + joined, not the original multi-line bytes from the reader.

Why: SSE frames span multiple lines (data: x\ndata: y\n\n). There's no single "raw line" to store. Reconstructing a single-line representation preserves the debugging intent without requiring storage of the full multi-line original.

Tradeoffs

Tradeoff Unbounded stderr buffer vs. fixed limit

Alternatives: (A) bytes.Buffer with no limit (current), (B) io.LimitReader wrapping stderr pipe, (C) stderr pipe + goroutine accumulator with size check.

Pros/Cons: (A) is simplest and matches spec's "zero deps" principle, but a misbehaving process could exhaust memory. (B/C) add complexity for a scenario that's unlikely in v1 (agent CLIs produce bounded stderr).

Winner: (A) — simplicity wins for v1. Can add a limit in a future phase if needed.

Tradeoff bufio.Scanner vs. bufio.Reader for frame scanning

Alternatives: (A) bufio.Scanner with 64KB initial / 1MB max buffer (current), (B) bufio.Reader + manual line reading, (C) json.Decoder streaming.

Pros/Cons: (A) handles line splitting and buffer growth automatically. (B) gives more control but more code. (C) only works for JSON, not SSE.

Winner: (A) — Scanner is purpose-built for this pattern, with configurable buffer limits. The 1MB max handles any realistic agent CLI frame.

Open Questions

Question Should Stderr() be safe to call while process is running?

The current Stderr() reads from bytes.Buffer that the subprocess writes to via cmd.Stderr. Calling it concurrently with active writes is technically a data race. The spec's concurrent-safety requirement covers stdin/stdout/close but not stderr reads.

Verify: Will any caller need to read stderr while the process is still running? If yes, we need a piped stderr + goroutine accumulator.

Question Is 1MB max scanner buffer sufficient for all providers?

Some LLM responses (especially tool_use with large file contents) could produce very large single-line JSON objects. The 1MB limit was chosen conservatively.

Verify: Test with real Claude Code / Codex output containing large tool results to confirm no scanner buffer overflow.
