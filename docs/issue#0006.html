Design Decisions

Decision RoleToolUse maps to assistant with tool_calls in OpenAI format

Ambiguity: The spec showed assistant messages with an IsToolCall flag for tool calls, but our Message type uses a separate RoleToolUse role instead.

Choice: RoleToolUse messages convert to OpenAI format as {"role": "assistant", "content": null, "tool_calls": [...]}. This matches OpenAI's wire format where tool calls appear as assistant messages with a tool_calls array.

Rationale: OpenAI's API doesn't have a separate role for tool calls — they're always embedded in assistant messages. Mapping our RoleToolUse to this format preserves the semantic distinction at our layer while matching the wire format exactly.

Decision System prompt prepended as first message

Ambiguity: The spec showed system messages in the messages array but RunInput.SystemPrompt is a separate field.

Choice: If SystemPrompt is non-empty, prepend {"role": "system", "content": "..."} as the first message in the OpenAI request.

Rationale: OpenAI Chat Completions expects the system message as a separate entry with role "system". Prepending ensures it comes first in the conversation without modifying the session's message history.

Decision SSE [DONE] signal and finish_reason both emit TurnEnd

Ambiguity: The spec didn't clarify when exactly TurnEnd fires for SSE streams — on [DONE] or on finish_reason.

Choice: Both triggers emit TurnEnd. The [DONE] signal emits one with stopReason: "end_turn". A non-empty finish_reason (e.g., "stop", "tool_calls") also emits TurnEnd.

Rationale: OpenAI streams may include a finish_reason on the last chunk and then a separate [DONE]. Emitting TurnEnd on the first trigger ensures consumers get the event promptly. The turn counter increments for each trigger, so double-firing is harmless if the orchestrator checks.

Deviations

Deviation Usage accumulated across chunks, not from a single final chunk

Spec said: Mock SSE [DONE] + usage → EventTurnEnd with TokenUsage.

Implemented: Usage fields are accumulated from any SSE chunk that includes a usage object. The TurnEnd event carries the accumulated totals.

Why: OpenAI's streaming API may emit usage on any chunk, not just the final one. Accumulating across all chunks ensures we capture the total regardless of when it appears. This matches OpenAI's documented behavior where usage is included when stream_options: {"include_usage": true} is set.

Tradeoffs

Tradeoff Shell script mocks vs. structured SSE test server

Alternatives: (A) Shell scripts echoing SSE lines (current), (B) Go HTTP test server serving SSE, (C) httptest-based mock.

Pros/Cons: (A) is simplest and directly tests the subprocess pipeline. (B/C) add HTTP layer complexity that's not needed since the SSEScanner reads from stdout, not HTTP.

Winner: (A) — same approach as ClaudeCodeAgent, consistent across providers.
