Design Decisions

Decision Used --mode rpc with JSONL protocol

Ambiguity: The issue said to research Pi CLI's protocol. Pi supports three modes: text (interactive TUI), json (single-shot), and rpc (interactive JSONL over stdin/stdout).

Choice: Used --mode rpc for full bidirectional JSONL communication. Reused the existing NewJSONRPCScanner from the process package since Pi's JSONL framing is identical to JSON-RPC line-delimited JSON.

Rationale: RPC mode provides the richest event stream (message_update, tool_execution_start/end, turn_end, agent_end) and supports multi-turn interaction. JSON mode is single-shot and doesn't stream. The JSONL framing matches our existing scanner perfectly.

Decision Single-shot prompt via prompt command

Ambiguity: Pi's RPC mode supports multiple commands (prompt, steer, follow_up, abort, etc.). It wasn't clear how to map a single Agent.Run() call.

Choice: Each Run() sends exactly one prompt command and reads events until turn_end or agent_end. Stdin is left open (not closed) since the RPC protocol expects the connection to stay alive.

Rationale: Matches the Agent.Run() contract — one call = one turn. The goroutine breaks the scanner loop after receiving turn_end, then proc.Close() handles subprocess cleanup.

Decision Simplified message passing — extract last user message only

Ambiguity: The spec's Section 6.3 said "TBD" for Pi message mapping.

Choice: Pi's prompt command takes a single message string. We extract the last user message from the session or use NewMessage.Content. No full history conversion is done.

Rationale: Pi manages its own conversation history internally via session files. We pass --no-session for ephemeral runs, so only the current prompt matters. Full history conversion would require understanding Pi's internal session format, which is an implementation detail.

Deviations

Deviation No full bidirectional message history conversion

Spec said: pi/convert.go should handle []Message ↔ Pi native format.

Implemented: convert.go only has messagesToPrompt() which extracts the last user message. No reverse conversion from Pi format to []Message.

Why: Pi's RPC protocol sends the entire conversation state in turn_end events, but parsing that into our Message format requires understanding Pi's internal message schema. For v1, we rely on the orchestrator to accumulate events into Messages. Pi manages its own session state internally.

Deviation Tool call arguments not captured during streaming

Spec said: Tool call events should include ToolInput.

Implemented: Tool calls from tool_execution_start don't include arguments — only toolCallId and toolName. The toolcall_delta events are silently skipped.

Why: Pi streams tool arguments incrementally via toolcall_delta events, similar to OpenAI's incremental tool_calls. Accumulating partial JSON deltas into a complete ToolInput would require a buffering layer. For v1, the tool name and ID are sufficient for the orchestrator to identify tool calls.

Tradeoffs

Tradeoff Reuse JSONRPCScanner vs. custom JSONL scanner

Alternatives: (A) Reuse process.NewJSONRPCScanner (current), (B) Create a new raw JSONL scanner.

Pros/Cons: (A) works because Pi's JSONL framing is line-delimited JSON, same as JSON-RPC. The scanner validates JSON and skips blank lines. (B) would avoid the "JSON-RPC" naming confusion but adds code for identical behavior.

Winner: (A) — zero additional code, proven scanner.

Open Questions

Question Should we support Pi's --session for persistent sessions?

Currently using --no-session for ephemeral runs. Pi has its own session management via --session-dir. If callers want Pi to maintain conversation state across Run() calls, we'd need to either: (a) let Pi manage sessions and pass --session <id>, or (b) rely on our own SessionStore as the orchestrator does.

Verify: Whether Pi's session persistence is needed for multi-turn flows.

Question Should toolcall_delta events be accumulated into complete ToolInput?

Currently tool calls arrive with just ID and name. Arguments stream as toolcall_delta events with partial JSON. Accumulating these would give full ToolInput but requires buffering logic in the parser.

Verify: Whether the orchestrator or caller needs complete tool arguments for approval decisions.
