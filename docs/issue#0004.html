Issue #4 — Registry — 2026-06-02

Design Decisions

Decision Get accepts opts for lazy instantiation

Ambiguity: The spec showed Get(name) but also mentioned provider-specific configuration (command path, API keys, model). It was unclear when/how config is passed.

Choice: Get(name string, opts ...AgentOption) — factory is called lazily on first Get, passing opts through.

Rationale: Registration is config-free (just name + factory). Configuration is supplied at Get-time via functional options, which is idiomatic Go. This avoids storing partial config at registration time and supports the same provider name with different configs in different contexts.

Decision Register takes explicit overwrite boolean

Ambiguity: The spec said "不允许重名注册" (duplicate registration not allowed) but didn't specify the API for handling attempts.

Choice: Register(name, factory, overwrite) with a boolean to allow intentional replacement. Returns ErrProviderAlreadyRegistered if duplicate and overwrite is false.

Rationale: A boolean parameter is explicit and discoverable. The caller consciously opts into overwriting. This supports testing (re-register mocks) and hot-reloading scenarios while protecting against accidental double-registration by default.

Deviations

Deviation Unregister returns error for nonexistent name

Spec said: "Unregister(name) 删除注册" — no error behavior specified.

Implemented: Unregister returns ErrProviderNotRegistered if the name is not found.

Why: Silent no-ops mask bugs in caller code (e.g., typos in provider names). Returning an error lets callers detect misconfiguration early. The cost is negligible — callers who don't care can ignore the error.

Tradeoffs

Tradeoff Stub factories vs. nil entries for built-in providers

Alternatives: (A) Real factory functions returning "not yet implemented" errors (current), (B) nil entries that Get must check, (C) no pre-registration — users register only what they need.

Pros/Cons: (A) lets callers discover available names via List() immediately and get clear errors. (B) is fragile — nil checks at every Get callsite. (C) removes discoverability entirely.

Winner: (A) — pre-registered stubs serve as documentation and provide actionable error messages ("claude agent not yet implemented") rather than opaque "not registered" errors.

None — implementation followed the spec as written.
