Issue #0025 — Context compression: define error types and compression contract — 2026-06-03

Design Decisions

Decision ContextLengthExceededError as a struct with Unwrap rather than a simple string error

Ambiguity: The PRD said "Add ContextLengthExceededError type implementing error interface" without specifying whether it should wrap underlying errors.

Choice: Defined ContextLengthExceededError as a struct with an Err error field and Unwrap() error method, enabling errors.Is and errors.As traversal.

Rationale: The original subprocess error (stderr output, exit code) carries diagnostic value. Wrapping preserves it while adding semantic type information. Callers can still access the root cause via errors.Is.

Decision IsContextLengthExceeded checks typed error first, then falls back to keyword matching

Ambiguity: The PRD specified "两者结合：优先用类型化错误，兜底用字符串匹配" but didn't specify whether both paths should be checked on every call.

Choice: errors.As is tried first (cheap, O(1) unwrap chain). If it fails, the full error message is lowercased and scanned for 5 known keywords.

Rationale: Type checking is always correct (no false positives). Keyword matching is a best-effort fallback for providers whose error formats we haven't yet typed. The short-circuit avoids unnecessary string operations when a typed error is already present.

Decision ContextCompressor placed in agentwrapper root package, not a sub-package

Ambiguity: The PRD said "interface lives in package agentwrapper" but said nothing about sub-package organization.

Choice: Defined ContextCompressor in the root agentwrapper package alongside Orchestrator, ApprovalHandler, and BudgetHandler.

Rationale: Consistent with existing patterns — all extension interfaces live in the root. A separate compressor/ sub-package would add import friction for the single-method interface that orchestrator directly consumes.

Deviations

None — implementation followed the PRD as written. All three deliverables (typed error, detection helper, Compressor interface) match US-001 and US-002 acceptance criteria exactly.

Tradeoffs

Tradeoff Hardcoded keyword list vs. configurable matchers

Alternatives considered: (A) Make the keyword list a parameter to IsContextLengthExceeded. (B) Allow callers to register custom error patterns via callback. (C) Hardcode a small, curated list.

Pros/cons: (A) Flexibility but complicates the function signature and every call site. (B) Maximum extensibility but over-engineered for a detection helper. (C) Simple, zero-config, covers 95%+ of real-world error messages from known providers.

Why (C) won: V1 scope explicitly says "不对非上下文超限的错误进行重试". Adding a config system for 5 strings before we've validated the approach is premature. If keywords need tuning, we change the list in one place.

Tradeoff "too long" keyword: coverage vs. false-positive risk

Alternatives considered: Include "too long" as a keyword vs. omit it.

Pros/cons: OpenAI and Gemini both use "too long" in their context-length error messages. But "too long" is also a common English phrase that could appear in unrelated errors (e.g., "request timeout: took too long").

Why included: The downside of a false positive is an unnecessary compression+retry cycle, which is cheap and idempotent. The downside of a false negative is a hard failure that could have been recovered. In harness engineering, prioritize recovery.

Tradeoff Single Compress method vs. Reduce/Expand pair

Alternatives considered: (A) Compress([]Message) []Message. (B) Reduce([]Message) []Message + Expand(summary, []Message) []Message pair.

Pros/cons: (B) enables two-phase compression where the orchestrator stores a compressed prefix and re-expands it later. This is useful for long-running sessions but requires session schema changes.

Why (A) won: V1 retry loop is stateless — compress, retry, succeed, done. No need to preserve compressed state across turns. A single method keeps the interface minimal and all compressor implementations trivial.

Open Questions

Question Should keywords be configurable per provider?

Assumption: The same 5 keywords work for all 4 agents (Claude, Codex, Pi, OpenCode). If one provider uses a completely different error format (e.g., "prompt_too_large" with underscores), IsContextLengthExceeded won't detect it.

Action: After #27 (agent detection) is implemented, verify that each provider's actual stderr output is covered by the current keyword list. If gaps exist, add keywords or make the list injectable via a variadic option.

Question Should "too long" be narrowed to reduce false-positive risk?

Assumption: "too long" alone is sufficient without qualifying context. A stricter alternative would be two-word pairs like "too long." immediately followed by "context" or "token" on the same line.

Action: Monitor for false positives during integration testing (#28, #29). If any non-context-length errors trigger compression, tighten the match to require co-occurrence with "context" or "token" in the same error message.

Question Should Compress accept context.Context for cancellation?

Assumption: Compressors are stateless and fast — sliding window is O(1) slice operations, naive summary is O(n) string slicing. No I/O, no LLM calls in V1.

Action: Keep the signature as-is for V1. If a future Summarizer calls an external LLM (per US-004 user-injected summarizer), add a context.Context parameter to the interface then (breaking change, but controlled by us).
