Design Decisions

Decision Package path sessionstore/memory/ instead of root

Ambiguity: The spec listed both session_memory.go at root and a session/memory/ package path in different sections.

Choice: Placed in sessionstore/memory/ as a standalone sub-package.

Rationale: Isolates the implementation from root-level types/interfaces. Future store implementations (e.g., SQLite, Redis) can live alongside memory/ without cluttering the root package. Avoids import cycles since root depends on types and this package also imports types.

Decision Used Go 1.21+ slices.Clone and maps.Clone for deep copies

Ambiguity: The spec's pseudocode used manual make + copy for deep-copying Messages.

Choice: Used slices.Clone(s.Messages) and maps.Clone(s.Metadata) (Go 1.21+).

Rationale: Project targets Go 1.26. Standard library helpers are clearer and less error-prone than manual copy loops. Message is a value-type struct (no pointers except json.RawMessage), so a shallow slice clone provides full isolation.

Deviations

Deviation Save replaces Messages instead of appending

Spec said: "Save() 原子更新 session，追加的 messages 不丢失"

Implemented: Save replaces the entire Messages slice with a deep copy of the provided session's Messages. It does not detect and append only new messages.

Why: The Get-Append-Save pattern is inherently racy at the application level (TOCTOU). Implementing merge/append semantics inside Save would require either: (a) a diffing algorithm that's fragile and complex, or (b) an AppendMessages method. The store provides atomicity and thread-safety; the caller is responsible for correct concurrent access patterns (e.g., serializing Get-Append-Save at the orchestrator level).

Tradeoffs

Tradeoff Two-level locking vs. single global lock

Alternatives: (A) Per-session RWMutex (current), (B) Single global RWMutex on the map.

Pros/Cons: (A) allows concurrent Save to different sessions without contention. (B) is simpler but serializes all writes globally. For 1-10 concurrent sessions (spec's expected load), the difference is negligible.

Winner: (A) — follows the spec's explicit pseudocode and scales better if session count grows.

Open Questions

Question Should Save return SessionNotFoundError for nonexistent sessions?

The current implementation returns *SessionNotFoundError if Save is called with a session ID that was never created or was deleted. The spec doesn't explicitly specify this behavior. An alternative is to silently create the session (upsert semantics).
