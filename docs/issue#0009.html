Implementation Notes

Issue #9 — Orchestrator — Turn 循环编排器 — 2026-06-02

Design Decisions

Decision Functional options pattern for Orchestrator configuration

Ambiguity: The spec defined ApprovalHandler and BudgetHandler as separate types but didn't specify how they're wired into the Orchestrator constructor.

Choice: Used NewOrchestrator(agent, store, opts...) with WithApprovalHandler and WithBudgetHandler functional options.

Rationale: Consistent with Go best practices for optional configuration. Both handlers are optional — nil approval means allow-all, nil budget means no limit. Functional options avoid constructor parameter sprawl if more hooks are added later.

Decision Approval deny produces synthetic ToolResult + continues turn

Ambiguity: The spec (5.2) says "deny → 构造 synthetic ToolResult → 作为 Message 追加到 session → 发送回 agent（继续 turn 循环）". But "发送回 agent" implies the orchestrator must re-inject the tool result into the agent's event stream, which isn't possible with a unidirectional <-chan Event.

Choice: On deny, the orchestrator appends a synthetic Message{Role: "tool_result"} to the session and emits both the ToolCall and synthetic ToolResult events to the caller. The agent continues its turn naturally — the real agent (e.g., Claude Code) will produce the ToolResult event itself since it executes tools internally. For agents like OpenCode that don't expose tool events, this path is moot.

Rationale: The orchestrator cannot inject events back into the agent's output channel. The real tool execution happens inside the agent subprocess. For approval deny, we can only prevent the tool from running by communicating with the agent through its native protocol — but none of the current agents support this. The synthetic ToolResult in the session ensures the session history is consistent, and the forwarded events inform the caller.

Decision Writeback on TurnEnd flushes assistant text + tool messages together

Ambiguity: The spec (5.3) lists separate writeback rules for TextDelta (accumulated) and ToolCall/ToolResult (individual), but the "写入时机" says "每个 TurnEnd 事件时，一次性追加该 turn 中累积的所有新消息".

Choice: Accumulate assistantText (from TextDelta) and turnMessages (tool_use + tool_result) separately during the turn, then on TurnEnd: append Message{Role: "assistant"} if text is non-empty, then append all turn messages, then save.

Rationale: This matches the spec's "一次性追加" and ensures session messages are in the correct order: assistant text → tool calls → tool results for the turn.

Deviations

Deviation Approval deny does not re-inject synthetic result back to agent

Spec said: "发送回 agent（继续 turn 循环）" — implies the orchestrator sends the denied tool result back to the agent process.

Implemented: The synthetic ToolResult is added to the session and forwarded to the caller, but not injected back into the agent subprocess. The agent continues its turn and produces its own ToolResult event (since tool execution happens inside the agent).

Why: The Agent.Run() interface returns <-chan Event — a read-only channel. There is no mechanism to write events back to the agent process. True "deny and continue" would require the agent to support a bidirectional protocol (e.g., writing a rejection to stdin), which none of the current agents implement. This is a fundamental interface limitation that should be addressed in a future iteration if needed.

Deviation Stop on end_turn/stop regardless of MaxTurns

Spec said: The state machine diagram shows a "STOP? maxTurns or stop" decision after writeback, implying the orchestrator checks both conditions.

Implemented: On TurnEnd with StopReason == "end_turn" or "stop", the orchestrator always returns immediately, regardless of MaxTurns. MaxTurns only stops the loop when the agent wants to continue (e.g., StopReason == "tool_use") but the turn limit is reached.

Why: When the agent says "end_turn", it has finished its response — there's no point forcing another turn. MaxTurns is a guard rail against runaway tool-use loops, not a mechanism to force additional turns after the agent is done.

Tradeoffs

Tradeoff Single goroutine event loop vs. separate event dispatcher

Alternatives: (A) Single goroutine reads from eventCh, processes events, and forwards to out channel; (B) Separate reader goroutine + dispatcher goroutine connected by an internal channel.

Pros of A: Simpler — no extra channel, no coordination. All state is local to one goroutine. No race conditions.

Pros of B: Could decouple reading from agent from writing to caller, potentially allowing backpressure handling.

Chose A: The orchestrator's state (assistantText, turnMessages, turnNumber) is inherently sequential — each event depends on the previous one. A single goroutine is correct by construction and avoids synchronization complexity.

Tradeoff Store.Save() on every TurnEnd vs. deferred save

Alternatives: (A) Call store.Save() after every TurnEnd writeback; (B) Save only when the orchestrator returns (deferred).

Pros of A: Session is persisted after each turn — if the process crashes, the last turn's state is recoverable.

Pros of B: Fewer writes; avoids partial state if writeback logic has a bug.

Chose A: The spec says "Save session to store" in the TurnEnd step. Per-turn persistence is the expected behavior for long-running agent sessions that may span many turns.

Open Questions

Question How should "deny" work with agents that execute tools internally?

Current agents (Claude Code, Codex, Pi) execute tools inside the subprocess. By the time the orchestrator sees a ToolCall event, the tool has likely already been executed. The approval check is therefore advisory rather than enforceable. Should we document this as a known limitation, or design a protocol where the agent waits for approval before executing?

Question Should the Orchestrator handle multi-Agent turns (tool_use stop → new Agent.Run call)?

When StopReason == "tool_use", the current implementation just continues reading events from the same eventCh. This assumes the agent handles tool execution internally and produces subsequent turns on the same channel. If a future agent needs external tool execution, the orchestrator would need to re-call Agent.Run() with the updated session. Should we design for this now?

Question Should the Orchestrator validate session message ordering?

The current implementation blindly appends messages without checking that tool_result messages match preceding tool_use messages, or that roles alternate correctly. Should we add validation, or trust the agent implementations to produce well-formed event streams?
