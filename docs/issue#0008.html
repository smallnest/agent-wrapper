Implementation Notes

Issue #8 — OpenCodeAgent — 2026-06-02

Design Decisions

Decision Non-interactive mode with single-shot event emission

Ambiguity: The spec (6.3) said "Pi Agent 和 OpenCode 的具体协议格式在实现 US-005/US-006 时调研确定" — the protocol was TBD.

Choice: Research confirmed OpenCode CLI has no streaming stdio protocol. Implemented opencode -p <prompt> -f json -q non-interactive mode, parsing the single-line JSON output {"response":"..."} and emitting one TextDelta + one TurnEnd event.

Rationale: This is the only machine-readable output mode OpenCode supports. The alternative (PTY + regex on TUI output) is far more fragile and version-dependent.

Decision Model passed via environment variable, not CLI flag

Ambiguity: Other agents use --model CLI flags; OpenCode has no such flag for non-interactive mode.

Choice: Pass model name as OPENCODE_MODEL environment variable. Pass system prompt as OPENCODE_SYSTEM_PROMPT env var.

Rationale: OpenCode's CLI doesn't expose a --model flag in non-interactive mode. Environment variables are the standard way to configure Go CLIs without explicit flag support.

Decision Binary search paths include ~/go/bin

Ambiguity: Other agents search PATH, ~/.local/bin, ~/.npm-global/bin. OpenCode is a Go binary.

Choice: Search PATH, ~/.local/bin/opencode, and ~/go/bin/opencode.

Rationale: OpenCode is installed via go install, which places binaries in ~/go/bin/. No npm path needed.

Deviations

Deviation No streaming — full response delivered as single TextDelta

Spec said: Section 6.3 implies adapting to a protocol similar to the other providers (streaming JSON-RPC/SSE).

Implemented: A single TextDelta event containing the entire response, followed by one TurnEnd. No incremental streaming.

Why: OpenCode's non-interactive mode blocks until the agent loop completes, then writes the entire response as one JSON object. There is no way to get streaming output without modifying the OpenCode source code.

Deviation No tool call/tool result events emitted

Spec said: Section 6.3: "复用 process/ 包中的 scanner（优先 JSON-RPC 或 SSE 格式）" — implies tool events should be parseable.

Implemented: Tool calls are executed internally by OpenCode but never exposed in the JSON output. The wrapper only sees the final text response.

Why: OpenCode's -f json output is {"response":"text"} only. Tool calls, thinking, and all intermediate steps are consumed internally and discarded.

Deviation No session history injection

Spec said: "agent 负责将历史转换为原生格式发给 CLI" — implies passing full message history.

Implemented: Only the last user message is extracted and passed via -p. No prior conversation history is injected.

Why: OpenCode's non-interactive mode accepts only a single prompt string via -p. There is no stdin-based message injection or session resume API.

Tradeoffs

Tradeoff JSON-RPC scanner reuse vs. simple stdout read

Alternatives: (A) Use process.NewJSONRPCScanner to read stdout line by line; (B) Use io.ReadAll to read entire stdout, then parse.

Pros of A: Consistent with other agents; handles multi-line output gracefully; process termination detection via scanner EOF.

Pros of B: Simpler code for a single-line output.

Chose A: Scanner-based reading is the established pattern in the codebase and handles edge cases (e.g., debug output mixed with JSON) more robustly.

Tradeoff Accept non-interactive mode vs. require protocol fork

Alternatives: (A) Ship with non-interactive mode limitations; (B) Fork OpenCode to add a JSON-RPC/SSE stdio protocol.

Pros of A: Works with stock OpenCode today; no maintenance burden of a fork.

Pros of B: Full streaming, tool visibility, session injection — matches other providers' capabilities.

Chose A: The spec explicitly anticipated this risk (11.2: "若无则用 PTY + 正则解析（降级方案）"). Shipping a limited but working provider is better than blocking on a fork. The agent interface allows upgrading the implementation later without changing callers.

Open Questions

Question OpenCode has been archived and renamed to Crush — should we track that?

The original opencode-ai/opencode repo is archived. The successor is Crush by Charm (charmbracelet/crush). If Crush adds a proper stdio protocol, the agent implementation should be updated. Should we open a follow-up issue to track Crush's protocol evolution?

Question Should OPENCODE_SYSTEM_PROMPT env var be documented as a convention?

The current implementation passes the system prompt via OPENCODE_SYSTEM_PROMPT, but this is not an official OpenCode feature — it's just an env var that may or may not be read by OpenCode's config system. Should we document this as a best-effort convention, or remove it to avoid false expectations?
