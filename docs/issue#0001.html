Issue #1 — Core Types — 2026-06-02

Design Decisions

Decision types/ 子包拆分

Ambiguity: SPEC 规划所有文件平铺在根包（Message、Session、Event、Provider 都在 package agentwrapper），未明确区分接口和纯类型。

Choice: 拆为两个包：package agentwrapper（Agent 接口、SessionStore 接口、ApprovalHandler、BudgetHandler）和 package types（Provider、Message、Session、Event、TokenUsage、所有 Error 类型）。

Rationale: 接口和纯类型分离后，provider 实现（claude/、codex/ 等）只需 import types 即可拿到全部数据模型，不依赖根包的编排逻辑。避免循环依赖 — provider 实现 Agent 接口，不需要 import 根包。

Decision Message 统一 struct 而非 per-role 类型

Ambiguity: 四种角色（user/assistant/tool_use/tool_result）字段差异大，可以用 4 个独立 struct 或接口。

Choice: 单一 Message struct，用 json:"omitempty" 按角色省略字段。提供 NewUserMessage、NewToolUseMessage 等构造函数。

Rationale: []Message slice 不需要类型断言或 switch，可直接 append。JSON 序列化出的格式与各 provider 的 wire format 转换更直接。每个角色一个 struct 会让 session 的 Messages 列表变成 []any，增加使用复杂度。

Decision UUID v4 用 crypto/rand 不用外部库

Ambiguity: 需要 UUID 生成，但 SPEC 要求零外部依赖。

Choice: 手写 20 行 newUUID()，用 crypto/rand.Read() + RFC 9562 的 version/variant 位操作。

Rationale: 零依赖的约束下这是唯一选项。google/uuid 或 gofrs/uuid 库是成熟方案但引入不必要的 transitive dependency。UUID 生成逻辑足够简单，自实现可接受。

Deviations

Deviation SessionStore 接口从 types 包移到根包

SPEC 说: SessionStore 接口和 Session 类型放在一起（session.go，根包）。

实际: SessionStore 留在 package agentwrapper 的 session.go，但引用类型来自 types/：Create() (*types.Session, error)。

Why: SessionStore 是行为接口而非数据模型。它的实现（MemorySessionStore）需要依赖根包或形成自己的子包。放在根包作为接口定义更自然 — 类似 io.Reader 在 io 包而非 types 包。对调用方来说 agentwrapper.SessionStore 比 types.SessionStore 更合理。

Tradeoffs

Tradeoff Event.Error 是 error 接口 vs 自定义 ErrorEvent struct

Alternatives: A) Event 包含 Error error 字段；B) Error 作为独立 event type 携带 ErrorCode string + ErrorMessage string。

Chosen: A. 调用方可以直接用 errors.As(evt.Error, &target) 做类型判断。代价：encoding/json 序列化 error 接口只会得到 null 或空对象 — 但 Event 通过 channel 传递，不需要 JSON 序列化。若未来需要持久化 Event 到 JSON，需要改 B。

Tradeoff ApprovalHandler 返回 *Decision 而非 Decision

Alternatives: A) 返回值类型 *Decision；B) 返回值类型 Decision。

Chosen: A. nil 指针有明确语义 — 等价于 allow。要是返回值类型，allow 时仍需构造一个空 Decision struct，无法区分"默认允许"和"显式允许"。代价：调用方需要 nil check。

Open Questions

Question Event.Error 在跨进程传输时如何处理？

Concern: 当前 Event 通过 Go channel 传递，error 接口能正常流转。但如果未来 Event 需要通过 JSON 序列化（如持久化日志、HTTP API 返回），error 接口的零值序列化行为会导致信息丢失。

可验证: 在实现 Orchestrator 或 HTTP 层时确认是否需要 MarshalJSON 自定义方法，或新增 ErrorString string 字段作为序列化兜底。

Question Session.Messages 并发安全策略？

Concern: Session struct 的 Messages 是普通 slice。调用方可能在 agent 运行时并发读写。当前的约定是"通过 SessionStore 写入"，但这只是文档约定，Go 编译器不强制。

可验证: 在实现 Orchestrator 和 MemorySessionStore 时确认并发测试覆盖。如需要，可以为 Messages 加 sync.RWMutex 或改为 immutable append pattern。
